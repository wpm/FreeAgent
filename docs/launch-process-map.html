Launch design — process map

Who starts what, who owns what, and how the processes talk. Dashed arrows = starts/owns; solid arrows = runtime communication. Browser zoom (⌘+ / ⌘−) or pinch.

Processes by lifetime

Reading by lifetime: NATS and the API live as long as the start/stop switch is on; the viewer server lives as long as the uv run collatz terminal session (Ctrl-C ends it); a worker lives for one episode; entities live inside the worker. Everything inside an episode talks only NATS; everything outside talks only REST; the API is the sole process that speaks both.

Not shown

The headless path: uv run freeagent-worker run collatz --episode-id … runs a worker directly from a terminal — same worker process, different parent; no API or viewer involved. And the future state: when the worker pool (ADR-0005, see process-map.html) lands, the API→worker spawn arrow is replaced by a solid arrow through a NATS work queue, worker daemons join the platform layer, the API stops needing apps installed, and it can move into docker — which kills the pid file.
